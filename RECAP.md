# Synelia Nexus — RECAP

> Chronological analysis of what I built in `/opt/data/synelia-nexus`
> on 2026-06-09. Generated by Olive after a crawl of the working tree,
> git history, kanban, and the deployment artefacts.

## TL;DR

In one session I took a fork of the Vercel **`chatbot`** template (a
single-user Next.js AI chat app) and turned it into a **multi-tenant
AI workspace product** called **Synelia Nexus** — the MVP for Synelia
Tech. Concretely I:

1. **Cut out the single-user scaffolding** — guest auth, document
   upload, voting, history API, model selector, suggestions, file
   uploads, the entire `(chat)/` route group, the MFE/demo plumbing,
   and the old Vercel AI Gateway.
2. **Replaced the auth model** — `better-auth` → `next-auth@5` with
   credentials + bcrypt. Members table + invite tokens, not anonymous
   guests.
3. **Replaced the chat API** — moved from per-chat stream routes to
   one `/api/chat` route that scopes every turn to a `workspaceId` +
   `threadId` pair, checks membership, applies a per-user rate limit,
   persists messages to SQLite, and streams the AI reply.
4. **Replaced the AI provider** — Vercel AI Gateway → `opencode serve`
   running locally on `:4096` (later deployed as a Swarm sidecar).
   Provider `opencode-go`, model `minimax-m3`.
5. **Added a real workspace filesystem** — `lib/ai/workspace-fs.ts`
   sandboxes every `workspace_read/write/list/edit` call inside
   `WORKSPACES_ROOT/<dirName>`, with `safeJoin()` that rejects
   absolute paths and `..` traversal. New workspaces get an
   `AGENTS.md` explaining the conventions.
6. **Added a scheduled-task subsystem** — Redis sorted set
   (`synelia:scheduled`) + `ScheduledTask` SQLite table + a standalone
   worker (`pnpm worker`, deployed as its own Swarm service) that
   polls every 5s, claims due tasks, replays the thread history, and
   runs the AI in a fresh turn.
7. **Built the multi-user UX** — home page lists workspaces and lets
   you create/join one; workspace page shows threads + members +
   artifacts + invite-token generator; thread page has a real chat
   client with `useChat()`, author bubbles, "AI is responding to X"
   indicator, tool-call transparency, and a stop button.
8. **Containerised it for Swarm** — multi-stage `Dockerfile` (deps →
   builder → runner, `tini` PID 1, non-root `nextjs` user), and a
   `docker-compose.yml` that deploys 4 services on Dokploy's overlay
   network: `app`, `worker`, `opencode` (sidecar with bind-mounted
   auth), `redis` (sidecar, AOF on). Traefik dynamic config in
   `traefik/synelia-nexus.yml` for HTTPS on
   `synelia-nexus.technocify.fr`.
9. **Documented it** — README, DEPLOY.md, .env.example, this RECAP,
   AGENTS.md, and a `.hermes/kanban.md` with the in-flight work.

Net: a 3.1.0 MVP you can clone → `pnpm install` → `opencode serve` →
`pnpm dev` → register two users → create a workspace → invite the
second user → chat with the AI in a shared thread. The AI can read &
write files in `/workspaces/<slug>/`, schedule future runs, and
persist its outputs as artifacts.

## The new data model

Drizzle schema, in plain English:

- **User** — NextAuth-shaped (`id`, `email`, `password`, `name`).
  Passwords are bcrypt-hashed; `Account`/`Session`/
  `VerificationToken` tables exist for the JWT callback.
- **Workspace** — `id`, `name`, `slug`, `description`, `dirName`
  (relative to `WORKSPACES_ROOT`), `createdById`.
- **WorkspaceMember** — `(workspaceId, userId)` PK, `role` ∈
  `{owner, admin, member}`.
- **WorkspaceInvite** — `id`, `workspaceId`, `token` (unique),
  `expiresAt?`, `maxUses?`, `uses`.
- **Thread** — `id`, `workspaceId`, `title`, `createdById`. Indexed
  by workspace.
- **Message** — `id`, `threadId`, `userId` (author for user msgs;
  triggering user for assistant msgs — this powers "AI is responding
  to X"), `role` ∈ `{user, assistant, system}`, `parts` (JSON, AI
  SDK v6 format), `attachments` (JSON). Indexed by
  `(threadId, createdAt)`.
- **Artifact** — `id`, `workspaceId`, `threadId?`, `createdById?`,
  `title`, `kind` ∈ `{text, code, sheet, image}`, `content`,
  `filePath` (relative to the workspace dir).
- **ScheduledTask** — `id`, `workspaceId`, `threadId?`, `createdById`,
  `prompt`, `runAt`, `status` ∈ `{pending, running, done, failed,
  cancelled}`, `lastError`, `resultMessageId`. Indexed by
  `(status, runAt)` and `workspaceId`.

Migrations live in `lib/db/migrations/`; the migrator is idempotent
and is re-run in a retry loop on container start
(`docker-compose.yml` `command:` for both `app` and `worker`).

        synelia-workspaces (volume)      │
        redis-data (volume)              │

The `app` service publishes port 3000 in `host` mode; Traefik
intercepts on 443. The `worker` service is a pure loop with no
healthcheck and no port. The `opencode` service bind-mounts
`/opt/data/.local/share/opencode/auth.json` from the host so the
sidecar can authenticate to the upstream model without baking the
credential into the image.

## What's likely to break next

- `opencode serve` has to be running before the chat route is
  called. The provider has `autoStartServer: false`, which is good
  (predictable lifecycle) but means a cold deploy with no
  sidecar healthcheck will 500. The Swarm healthcheck on
  `opencode` covers this; just don't blow it away.
- The `worker` service runs `migrate.ts` in a retry loop. SQLite
  under multi-process access is fine for this scale, but the two
  processes both calling `Drizzle.migrate()` at boot means a brief
  lock. The retry loop handles it; don't remove the `sleep 2`.
- The chat client currently doesn't poll for new messages from
  other users. In a real multi-user session you'd see your own
  messages stream, but the other user has to refresh. The GET
  endpoint with `?since=` is there; the consumer is not. That's the
  single biggest UX gap.
- No e2e test for the multi-user flow. Register → workspace → invite
  → join → chat. It would have caught the migration lock and the
  `safeJoin` edge cases.

## Decisions worth remembering (for next time)

1. **Don't fork the entire Vercel chatbot.** Too much Vercel-flavoured
   scaffolding (geo, MFE config, demo flags, Vercel Analytics, token
   usage, etc.) to carry. Either use AI SDK examples directly or
   start from `create-next-app`. ~25% of the work on 2026-06-09 was
   deletion.
2. **Workspaces as real folders is the right call.** The audit log
   is `git log workspaces/<slug>/`. The AI's "memory" is just files.
   This composes with every other tool the user already has.
3. **Schedule tasks in Redis, persist in SQLite.** Redis is the
   queue, SQLite is the journal. The worker is dumb — it just polls.
4. **Sidecar the LLM gateway.** Putting `opencode serve` in its own
   container means the auth credential lives on a host bind-mount,
   not in the image. Swapping providers is a config change, not a
   rebuild.
5. **One chat route, two query params.** `/api/chat?workspaceId&threadId`.
   Don't grow the API surface — every workspace/thread combo
   composes through the same handler. The `buildWorkspaceTools()`
   factory takes the IDs and scopes the tools.

  strctier follow cowork html decisions, components, close like pixel
  replicate"**. So we port the Cowork UI into the existing Next.js shell
  pixel-for-pixel, keeping the deeper auth + DB + worker substrate.

  (Conversations / Artefacts / Connaissances / Routines / Équipe) is the

- **The Claude-cowork-style project Conversations tab** — big composer

- **7 modals** — `NewProjectModal` (no icon, no color — palette picks
  automatically), `InviteModal`, `NewChatModal`, `NewPromptModal`,
  `UsePromptModal`, `VisibilityModal`, `ArtifactModal` +

- **Realtime "en direct" indicators** are the heart of the product. Live
  AI streaming, steering (interrupt + re-orient), copresence banner,
  ghost typing, live artifact build (the risk matrix fills row by row as

- **No "en ligne" presence chrome** — stripped per the user in chat1.
  Topbar counter, sidebar presence block, green dots on avatars, "en
  ligne" mentions — all gone. "En direct" stays because it's the

### Gotchas (recorded so we don't trip on them)

- CSS prefix `ad-` collides with ad-blockers in preview envs. Admin
  console must use `adm-`.
- `SYN.RISK_ROWS` must be on the global SYN object before the chat reads
  it, or the stream loop throws.
- Modal state leaks between artifacts unless you set `key={artifact.id}`.
- `cancelStream` for chat interrupt must be a `useRef`, not state.
- No emoji in the UI. No gradients. Violet-tinted shadows only. French
  copy with vouvoiement throughout.
